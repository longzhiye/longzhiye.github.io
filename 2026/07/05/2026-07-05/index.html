一、什么是 GMS 认证？

GMS 认证（Google Mobile Services Test Certification）是由 Google 面向全球 Android 硬件厂商提供的官方兼容性认证体系。其目标是确保基于 Android 开发的硬件产品，能够与 Google 提供的系统服务、预装应用及 GMS 套件稳定、合规地协同工作。

为什么 GMS 认证不可或缺？

价值维度 说明
兼容性与稳定性 确保 Android 系统产品符合 Google 兼容性标准，减少碎片化问题
GMS 服务授权 获得合法使用 GMS 服务及 Google Play Store 的资格
品牌与市场准入 取得 Android Logo 使用授权，产品可进入海外市场销售
应用生态保障 确保设备上 Android 应用正常运行，并抵御恶意软件侵害

二、GMS 认证完整流程

GMS 认证并非单一测试环节，而是涵盖法律协议签署、产品送检、第三方实验室测试及 Google 授权的完整链路。

四步认证流程

关键术语说明

  • MADA（Mobile Application Distribution Agreement）：移动应用分发协议，授权设备预装和分发 Google 应用
  • AFA（Anti Fragmentation Agreement）：反碎片化协议，约束厂商不得破坏 Android 兼容性
  • 3PL（Third Party Lab）：第三方认证实验室，主要提供软件认证、品牌认可、GMS 需求对接及预测试技术支持

三、认证项目体系：GMS Express 家族

Google 针对芯片厂商与 ODM/OEM 建立了分层认证项目，三者必须全部执行。

三大认证项目对比

项目 核心要求
GMS Express Google 与芯片厂商合作完成验证的 Turnkey 基准版本；芯片厂商向 ODM 分发完整软件包（BSP + 经 Google 认证的 Android 软件及 GMS 套件）；准许设备厂商直接接收 Google 月度安全更新
GMS Express Plus 必须使用 GMS Express Build；核心程序 Category 的 default intents 不可被覆盖；浏览器以外 Category 不得预装同类应用
GMS Express 2.0 GMS Express 变体；SoC 构建的部分以二进制发布，其他分区以源代码形式发布

软件交付链路

GMS Express Plus 认证费用报销

对于支持原生 GMS Express Plus 的设备，在通过 3PL 认证后，若满足报销条件，Google 将报销认证费用：

  • 报销范围：覆盖所有设备的 MR（维护版本）和 SMR（安全维护发行版本）
  • 核心收益：
    • 100% 设备通过 Google 认证
    • 缩短约 30% 认证时间
    • 显著提升用户体验

四、认证测试套件详解

GMS 认证涉及大量自动化与半自动化测试套件，覆盖兼容性、安全性、性能及固件分析等多个维度。

核心测试套件一览

测试套件 全称 说明
CTS Compatibility Test Suite 兼容性测试套件（自动化）
CTS Verifier — CTS 补充，需手动验证
CTS on GSI — 与 CTS 使用同一测试套件，运行于 GSI 环境
VTS Vendor Test Suite 供应商测试套件（Treble 架构）
GTS GMS Test Suite GMS 测试套件
GTS Verifier — GTS 补充，手动验证（Android 13）
GTS Interactive — GTS 补充，半自动化（Android 14+）
GTS Root — GTS 补充（Android 15+）
STS Security Test Suite 安全测试套件
Firmware Analysis — 固件版本分析（原 BTS）
APTS Android Performance Test Suite 安卓性能测试套件
AACT Android Auto Connectivity Tests 安卓自动连接测试（2024 新增）
Connectivity Test — 连接测试，含 GtsBetocqAqtTests（2026/Android 17 新增）

测试套件演进时间线

年份 变更
2021 新增针对 Go 设备的性能测试套件，取代 GOATS
2022 新增 Firmware Analysis，取代 BTS（Build Test Suite）
2023 新增 GTS Interactive，逐步取代 GTS Verifier
2024 新增 GTS Root 及 AACT 测试套件
2026 新增 Connectivity Test（Android 17 平台）

五、豁免流程（Waiver）

Google 的百万级测试用例中，部分案例可能存在测试代码本身的 Bug。对此，Google 提供了正式的豁免流程（Waiver）。

豁免适用场景

  1. 芯片厂商基准版本已通过认证：若客户对基准版本进行修改导致测试失败，经调查确认是 Google 测试代码问题，可向 Google 提交 Issue 和 Report 申请豁免
  2. 最新测试包已通过：若失败案例在 Google 发布的最新测试包中已通过，直接提交测试通过报告至 APA，无需单独申请豁免

六、附录：关键术语速查

缩写 全称 说明
IR Initial Release 初始发布版本
MR Maintenance Release 维护版本
SMR Security MR 安全维护发行版本
EMR Emergency MR 紧急维护发行版本
Treble — 包含 VTS 和 CTS on GSI
MADA Mobile Application Distribution Agreement 移动应用分发协议
AFA Anti Fragmentation Agreement 反碎片化协议
BCR Biometric Report 生物识别报告
Representative Product SKU — 经批准、具有完整测试报告集、不依赖其他代表性构建的构建版本
Derived Product SKU — 使用相同 SoC，声明相同或 android.hardware.* 子集特性，并设置 ro.build.representative 属性指向代表性版本的派生构建

Derived Product SKU 判定条件

一个构建被认定为 Derived（派生），需同时满足：

  1. 使用相同的 SoC
  2. 声明相同或子集的 android.hardware.* features
  3. 设置 ro.build.representative 属性值为代表性构建的 fingerprint

七、工程师实战建议

认证前准备清单

  • 确认已与 Google 签署 MADA、AFA 协议
  • 基于芯片厂商 GMS Express Turnkey 基准版本进行开发，避免破坏 default intents
  • 在 userdebug 版本上完成 STS 等调试向测试
  • 在 User/GSI 版本上完成 CTS、GTS、VTS 等正式认证测试
  • 提前整理产品本体、外包装、说明书送检材料

常见踩坑点

  1. 修改核心 Category 的 default intents — 将导致 GMS Express Plus 资格丧失及报销失效
  2. 预装同类 Category 应用 — 浏览器以外 Category 不得有重复预装
  3. Derived 构建未正确设置 ro.build.representative — 将无法享受 Representative 构建的测试减免
  4. 忽略 Android 版本对应的测试套件变更 — 如 Android 14 需用 GTS Interactive 替代 GTS Verifier
  5. SMR 版本遗漏 GTS-SMR 测试 — Derived SMR 构建需额外执行 GTS-SMR
